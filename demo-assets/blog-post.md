# OpsTwin Recovery Arena

**An OpenEnv environment for training LLM agents on enterprise incident response.**

Built for the Meta PyTorch OpenEnv Hackathon Grand Finale (Bangalore, April 25 to 26, 2026) under Theme 3.1 (World Modeling, Professional Tasks) by Shashwat Rajan and Tanish Shitanshu.

## The problem

Every benchmark of enterprise agents (WorkArena, CRMArena-Pro, World of Workflows) shows the same failure mode. The model can handle single-step API calls, but as soon as a task has **hidden state**, **multi-step dependencies**, or **policy constraints**, accuracy collapses. The industry is racing to train agents on real work, but there is no good environment for the hardest piece of that work: incident response, where the visible alert is usually the wrong thing to fix.

## What OpsTwin does

OpsTwin simulates a production outage in progress. The agent plays the on-call incident commander. In any given episode:

- **Five operational desks** (Incident Command, SRE, Security, Support, Release) expose different command sets and filtered views of the world. Cross-desk coordination is structural.
- **Hidden state** drives every reward. The agent cannot see the real root cause, the full blast radius, stale approval flags, or actual feature flag values until it runs the right inspection action.
- **Rewards are code**, not an LLM judge. Six pure deterministic functions (service recovery, customer outcome, security compliance, change hygiene, communication quality, operational efficiency) combine into the final score.

Three hand-authored scenarios stress-test different failure modes:

1. **Bad Release.** Root cause is a feature flag, not the deploy. Rolling back symptomatically fixes the outage but cascades into an unrelated analytics service through a hidden edge. The correct move is `VERIFY_FLAG` then `FLIP_FLAG`.
2. **Security CVE.** Patch exists but its approval is 48 hours stale. Quarantining the affected service is fastest but breaks the VIP SLA and triggers legal notification. The correct path is `APPROVE_EXCEPTION` then `RERUN_PIPELINE`.
3. **Data Quality Cascade.** Infrastructure fix is necessary but not sufficient. Customer comms are a required issue. Agents who fix the ETL and call `DONE` cap at 0.65 instead of 0.92.

Plus a stretch **False Positive Trap**, where aggressive rollback is the wrong answer.

## Training evidence

We fine-tuned Qwen3-1.7B on 3,600 expert-trajectory examples generated by our rule-based solver. Training ran 16 minutes on an RTX 5090.

| Scenario | Random | Untrained Qwen3-1.7B | **Trained** | Heuristic (upper bound) |
|---|---|---|---|---|
| bad_release | 0.24 | 0.18 | **0.42** | 0.99 |
| security_cve | 0.22 | 0.20 | **0.94** | 0.99 |
| data_pipeline_regression | 0.24 | 0.20 | **0.75** | 0.99 |
| false_positive | 0.24 | 0.85 | **0.84** | 0.99 |
| **Average** | **0.24** | **0.36** | **0.74** | **0.99** |

The trained model more than doubles the baseline. The largest delta is `security_cve` (0.20 to 0.94), which requires a five-step policy (assess blast radius, refresh stale approval, rerun pipeline, prioritize VIP, draft comms). Training data: [Deltasthic/opstwin-qwen3-1.7b-sft](https://huggingface.co/Deltasthic/opstwin-qwen3-1.7b-sft).

The environment is a drop-in replacement for any OpenEnv client. Point your TRL trainer or agent loop at the Space URL and start training.

## What's next

The V1 weakness is `bad_release` at 0.42. The model hallucinates feature flag names instead of copying them from the runbook. Our V3 branch adds wrong-name augmentation and a dedicated `VERIFY_FLAG` curriculum. V3 numbers land in time for the Grand Finale demo.

If you want to push this further with us, the environment supports procedural scenarios out of the box (see `server/generator.py`). New scenario families take about an hour to author. The bottleneck is good ideas, not code.
